Ornge Project Documentation

Project Overview:

We are developing our web application using JavaScript technologies, including React.js and Node.js. Our core application development is complete, and we are now focused on enhancing existing features and accommodating change requests. Ornge is primarily designed for the 3D printer company Formlabs, but after a successful integration, we are now integrating with another major 3D printer company, SprintRay. Our goal is to generalize the platform to minimize development for future integrations. We manage development using Kanban.

Project Details:

Ornge has both a public and a private web app. The public app serves as a CMS, showcasing what Ornge offers, while the private app, which requires signup and login, handles the core functionality. Doctors can access various services after logging in.

Different Components of the Project:

  • Frontend codebase for the private portal (React)
  • Frontend codebase for the public portal (Gatsby)
  • Frontend codebase for UI (React-styled components)
  • Backend codebase for the private portal (Node.js)
  • Backend codebase for the public portal (Strapi)

Project Environments:

We have staging and production environments for both the private and public applications.

New Developer Onboarding:

Tools Needed:

  • Visual Studio
  • Node.js (latest stable version preferred)
  • Redis (latest stable version preferred)
  • Global Protect (optional)

Development Machine Setup:

Ensure the tools listed above are installed. Set up Git using the following commands:

git config --global user.name "Your Fullname"
git config --global user.email "Your Email address"

For GitHub access, refer to GitHub SSH setup instructions. Clone the application:

git clone https://github.com/ornge/private-backend.git
git fetch && git checkout development

Create a .env file and populate it with the required configurations.

Building the Application:

Run the following commands to build the app:

yarn install
yarn start

Access the application at http://localhost:3000.

Accounts:

For new team members, access is needed for:

  • Jira: Task management
  • GitHub: Repository access
  • AWS: Optional for service manipulation
  • Ornge: Account creation on qa.ornge.com (e.g., test@ornge.com with password dev@12345678)

Development Process:

The application uses a modified feature model for development. Developers work in feature branches, with the development branch as the base. For hotfixes, a hotfix branch is created from the master branch. Below is a sample folder structure for development:

|-- arbisoft
    |-- ornge
        |-- private-backend
        |-- private-frontend
        |-- public-cms-backend
        |-- public-cms-frontend
        |-- ui

Commit Template and Branches:

Set up the local Git commit template with the following format:

:gitmoji: <title>

<empty-line>

Ticket: https://ornge.atlassian.net/browse/ORNGE-000

Example commit:

:sparkles: ORNGE-123 implements new feature

Branch names follow this format:

<dev-name>/ticket-no-one-liner

Example:

haniya/ORNGE-111-fixes-something

Technologies:

The following technologies are used:

  • React.js (v16.9) for client-side development
  • Node.js (>=12) for server-side
  • PostgreSQL for the database
  • Redis (v4.5) for in-memory data storage

Non-functional Requirements:

  • Performance: Response time optimized for orthodontists in the US using 3D printers (Formlabs, SprintRay). Target response time: 2 seconds.
  • Auditability: User-facing error messages and logs are maintained in error.log and exceptions.log.
  • Reliability: High reliability with a recovery time of less than a minute during redeployment.
  • Usability: Designed for ease of use, primarily for doctors using desktop and mobile platforms. Supported browsers include Chrome, IE, and Firefox, and devices range from 1920×1080 desktop screens to 414×896 mobile screens.

Code Review Process:

At Ornge, code reviews ensure code quality. Reviewers focus on functionality, complexity, tests, naming conventions, comments, and documentation. Pair programming and in-person reviews are encouraged.

Release and Deployment:

Ornge follows a straightforward release and deployment process. Releases are documented on GitHub, and deployments occur as needed, especially during maintenance phases.
